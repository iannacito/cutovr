# Demo setup &mdash; Cutovr

This document explains how to run a repeatable Cutovr demo (tens to
hundreds of times) against a single dedicated QuickBooks Online demo
company **without** purging QBO between runs and **without** flipping
between QBO accounts.

The product side of this workflow lives in `demo_mode.py` and the
`/demo` page. This file describes the deployment side.

---

## Recommended topology

Run **two** Render services, both pointed at this repo:

| Service        | Hostname                                | Purpose                          | `DEMO_MODE` | `QBO_ENVIRONMENT`        |
|----------------|------------------------------------------|----------------------------------|-------------|--------------------------|
| Production     | `www.pclawmigrate.com`                   | Real customers, real QBO data    | unset / `0` | `production`             |
| Demo / staging | `demo.pclawmigrate.com` (Render staging) | Repeatable demos, sandbox data   | `true`      | `sandbox` *or* dedicated `production` company |

Keep them on **separate Render services**. The demo service must not
share the SQLite disk with production (otherwise demo resets would
churn production audit rows).

---

## Demo service env vars

Set these in the Render dashboard for the demo service (Settings →
Environment). **Do not commit secrets to this repo.** Anything labelled
"set in dashboard" must be pasted into Render directly.

```text
APP_ENV=production           # production-grade cookie/CSRF hardening even on demo
DEMO_MODE=true               # unlocks the /demo page and demo controls
QBO_CLIENT_ID=<set in dashboard>
QBO_CLIENT_SECRET=<set in dashboard>
QBO_REDIRECT_URI=https://demo.pclawmigrate.com/oauth/callback
QBO_ENVIRONMENT=sandbox      # OR `production` if using a dedicated demo company
QBO_REAL_IMPORT=1            # the demo flow needs real QBO writes to be impressive
ENCRYPTION_KEY=<auto-generated by Render>
SECRET_KEY=<auto-generated by Render>
IMPORT_HISTORY_DB=/var/data/import_history.sqlite3
APP_DB=/var/data/app.sqlite3
OPERATOR_EMAILS=dan@yourfirm.com   # optional; lets you see demo on prod too
```

The `OPERATOR_EMAILS` line is optional. If set, the listed operators
will *also* see the `/demo` link on the production service, which is
occasionally useful for support / debugging but should be left empty in
the normal case.

### Intuit settings

In the Intuit developer dashboard:

1. Add `https://demo.pclawmigrate.com/oauth/callback` to the **Redirect
   URIs** list for the same app you use in production (sandbox keys), or
   create a separate Intuit app whose keys you put on the demo service.
2. Decide which QuickBooks Online company is your **dedicated demo
   company**. Bookmark it. Always connect the demo service to that
   company &mdash; never to your real firm.
3. If the wrong QBO company is ever connected, click **Disconnect** on
   the `/quickbooks` page, then **Connect** and pick the demo company in
   Intuit's company chooser. We do not (and cannot) force Intuit's
   chooser beyond the normal OAuth flow.

## One-time setup checklist

1. Sign up on the demo service as `demo@yourfirm.com` (or whichever
   email you want to demo from).
2. Visit `/demo`. You should see the demo workspace page with a
   **Start a new demo** button.
3. Click **Start a new demo**. This mints a demo run id and writes an
   audit row. (Nothing has happened in QBO yet.)
4. Connect QuickBooks &mdash; pick the dedicated demo QBO company in
   the Intuit chooser.
5. Confirm the **Connected QuickBooks company** panel on `/demo` shows
   the correct realm id and name.

## Repeat-demo loop

For every subsequent demo:

1. On `/demo`, click **Start a new demo**. The app archives the
   previous demo's jobs and mints a fresh run id.
2. Download the four bundled sample reports from the same page (COA,
   Trial Balance, General Ledger, Trust Listing). Their transaction
   ids / customer / vendor names embed the new run id so the import
   does not collide with prior demo imports against the same QBO
   company.
3. Walk the customer through the normal flow (`/dashboard` → upload →
   COA review → opening balance → import → verify).

### Why this avoids the duplicate-import block

- Each demo run mints a unique run id like `D-20260522T143012-3F9A`.
- Sample GL rows carry the run id in the transaction id (e.g.
  `JE-20260522T1430123F9A-0001`), in memos, and in customer / vendor
  names.
- QBO's DocNumber-uniqueness check (and the app's own sha256 file
  dedup) sees each run as a fresh payload, so it does not reject the
  import as a duplicate.

### What still requires manual cleanup

- This module does **not** purge QBO data. Trust journal entries,
  customers, vendors, and accounts created by each demo accumulate in
  the connected QBO company. If you want to start fresh on the QBO
  side, you must do that inside QuickBooks (e.g. revoke the connection
  and provision a fresh sandbox / new dedicated demo company).
- The recommended cadence is to run a single dedicated demo company
  for a long time and accept that the demo data builds up &mdash; the
  app-side reset is what keeps the demo experience clean from the
  customer's point of view.

## What the production service should look like

To keep this honest, here is what production should **not** have set:

```text
DEMO_MODE  # unset (or `0`)
APP_DEMO_MODE  # unset (or `0`)
```

With those unset, `/demo`, `/demo/start`, and the demo sample CSV
routes all return 404 to normal logged-in customers, and the **Demo**
nav link is hidden. (Operators in `OPERATOR_EMAILS` see it
unconditionally &mdash; that's intentional.)

## After merging this PR &mdash; what Dan still has to configure

The code is in this PR. Render and Intuit still need a few clicks:

1. **Render &mdash; create a demo service** pointing at the same repo
   with the env vars listed above. Easiest path: clone the existing
   service and override `DEMO_MODE=true`, `QBO_REDIRECT_URI`, and (if
   different) `QBO_CLIENT_ID` / `QBO_CLIENT_SECRET`.
2. **Render &mdash; attach a fresh disk** to the demo service so its
   `/var/data/app.sqlite3` and `/var/data/import_history.sqlite3` are
   isolated from production.
3. **Intuit dashboard &mdash; add the demo redirect URI** to whichever
   app keys the demo service uses. Required for the OAuth callback to
   exact-match.
4. **Pick the dedicated demo QBO company** (sandbox or a real,
   throwaway production company you control) and bookmark it.
5. **Sign up on the demo service** as the demo user, visit `/demo`,
   click **Start a new demo**, connect QuickBooks, and pick the demo
   company in Intuit's chooser. Verify the connected company name and
   realm id on `/demo`.
6. **Smoke test** &mdash; download all four sample CSVs, upload them on
   `/dashboard`, walk the end-to-end flow once, and confirm that
   running **Start a new demo** + re-uploading the new sample CSVs
   posts cleanly to QBO a second time without duplicate blocks.
