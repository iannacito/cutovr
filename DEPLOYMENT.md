# Deployment Guide (Render)

This guide covers a production-beta deployment of pclaw-qbo-v2 on
[Render](https://render.com). Stick with sandbox until you've completed the
Intuit production review.

## 1. Prerequisites

- A Render account.
- An Intuit developer account at <https://developer.intuit.com/>.
- One sandbox app already created in the Intuit developer dashboard. Note
  its **Client ID** and **Client Secret**.

## 2. Push the code to a Git repo

Render deploys from Git. Create a new GitHub/GitLab repo, push this folder,
and remember the repo URL.

```bash
git init
git add .
git commit -m "Initial pclaw-qbo-v2 deploy"
git branch -M main
git remote add origin <your repo URL>
git push -u origin main
```

## 3. Create the Render service from the blueprint

1. In Render: **New** → **Blueprint** → connect the repo.
2. Render will read `render.yaml` and create one Web Service plus a 1 GB
   persistent disk mounted at `/var/data` (used for the SQLite import
   history database).
3. The first deploy will fail until you fill in env vars — that's expected.

## 4. Set environment variables (Render Dashboard → service → Environment)

| Variable | What to put | Notes |
| --- | --- | --- |
| `QBO_CLIENT_ID` | from Intuit developer dashboard | Sandbox app's Client ID |
| `QBO_CLIENT_SECRET` | from Intuit developer dashboard | Treat as a secret |
| `QBO_REDIRECT_URI` | `https://<your-render-url>/oauth/callback` | Exactly matches Intuit Redirect URI below |
| `QBO_ENVIRONMENT` | `sandbox` | Switch to `production` only after Intuit review |
| `QBO_REAL_IMPORT` | `0` initially, then `1` | Demo-mode safety gate |
| `ENCRYPTION_KEY` | auto-generated by render.yaml | Fernet 32-byte url-safe base64 |
| `SECRET_KEY` | auto-generated by render.yaml | Flask session signing — **required** when `APP_ENV != local` |
| `APP_ENV` | `production` | Enables Secure cookies and refuses to start without `SECRET_KEY` |
| `IMPORT_HISTORY_DB` | `/var/data/import_history.sqlite3` | On the persistent disk |
| `APP_DB` | `/var/data/app.sqlite3` | On the persistent disk |

Both `ENCRYPTION_KEY` and `SECRET_KEY` use Render's `generateValue: true`
in `render.yaml`, so they're populated on first deploy. Keep them stable —
rotating `ENCRYPTION_KEY` invalidates any encrypted file/token already at
rest.

## 5. Add the Redirect URI in Intuit

1. Open <https://developer.intuit.com/> → **My Apps** → your sandbox app.
2. Go to **Keys & OAuth**.
3. Under **Redirect URIs**, add:

   ```
   https://<your-render-url>/oauth/callback
   ```

   (Must match `QBO_REDIRECT_URI` exactly — no trailing slash, http vs
   https matters.)

4. Save.

## 6. Trigger a deploy and smoke test

1. In Render: **Manual Deploy** → **Deploy latest commit**.
2. Once status is **Live**, open your service URL.
3. You should be redirected to `/login`. Click **Create one** to sign up
   the first firm account.
4. Upload `test_data/02_general_ledger.csv`.
5. Click **Connect to QuickBooks** → consent in the correct sandbox.
6. The job page should show **Company:** with your sandbox name.
7. With `QBO_REAL_IMPORT=0` (still demo): click **Import** — confirm
   you see the demo flash.
8. Set `QBO_REAL_IMPORT=1` in Render env vars and redeploy.
9. Click **Import** — expect the green flash listing 5 created JEs and an
   automatic Verification panel showing OK ✓.

### Pre-deploy smoke checklist

Before flipping `QBO_REAL_IMPORT` on production:

- [ ] Render service is **Live** with the persistent disk mounted at `/var/data`.
- [ ] `APP_ENV=production` — confirms session cookies are `Secure`.
- [ ] `SECRET_KEY` is set in Render Environment (not blank).
- [ ] `ENCRYPTION_KEY` is set in Render Environment (rotating it
      breaks all stored encrypted tokens — leave alone after first deploy).
- [ ] `QBO_REDIRECT_URI` exactly matches the Redirect URI registered in Intuit.
- [ ] Cloning a private window: signup → login → upload → connect → demo
      import all work without console errors.
- [ ] Submitting a form with the page open in two tabs and stale CSRF token
      shows the friendly "session expired" flash, not a 500.

## 7. After a successful smoke test

- Re-uploading the same file or any of the same `transaction_id` values
  triggers the duplicate guard and refuses to post.
- The import history table on the job page is read from
  `/var/data/import_history.sqlite3`, which survives redeploys.

## 8. Production mode (later)

Going to `QBO_ENVIRONMENT=production` requires Intuit's review of the app
(see <https://developer.intuit.com/app/developer/qbo/docs/develop/get-started-with-the-quickbooks-online-api/move-from-sandbox-to-production>).
You will also want to:

- Switch SQLite for managed Postgres (see `PHASE_2_NEXT_STEPS.md`).
- Move uploaded encrypted CSVs off the local disk (S3 + KMS).
- Add user authentication and per-tenant isolation.

## Rolling back a bad import

If a run created bad JEs in QBO:

1. In QBO, go to the company gear → Audit Log → filter by user — find each
   `JournalEntry` and **Delete** it.
2. Re-uploading the same CSV will be blocked by duplicate protection. Use a
   different file or, for testing only, manually `DELETE` the matching row
   from the `imports` table:

   ```bash
   sqlite3 /var/data/import_history.sqlite3 \
     "DELETE FROM imports WHERE id = <import id from job page>;"
   ```

The schema has `ON DELETE CASCADE` on the child tables, so transaction and
entity rows go with it.
